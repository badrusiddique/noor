# Android Keystore — Backup & Recovery

> **CRITICAL:** Lose the upload keystore = you can never update the Android app. **Ever.** Google Play does not allow re-uploading apps signed with a different key. If this file is lost or compromised, the only recovery path is publishing a new app under a new package name and asking every user to migrate.

This document is the single source of truth for how Noor's Android signing key is generated, stored, backed up, rotated, and recovered.

---

## 1. Generation

Generated **once** at Phase 0 (this phase), stored locally during Phase 8b, used for the first signed APK at `v1.0.0`.

```sh
keytool -genkeypair -v \
  -keystore noor-upload.keystore \
  -alias noor-upload \
  -keyalg RSA \
  -keysize 4096 \
  -validity 10000 \
  -dname "CN=Badrudduza Siddique, OU=Noor, O=Noor, L=Mumbai, ST=Maharashtra, C=IN"
```

When prompted:

- **Keystore password:** generated by 1Password (32 chars, mixed). Store in 1Password vault `Noor — Production Secrets` under entry `Android Upload Keystore`.
- **Key password:** same as keystore password (recommended for upload-only keys; simpler recovery story).

Record the **SHA-256 fingerprint** for later verification:

```sh
keytool -list -v -keystore noor-upload.keystore | grep "SHA256:"
```

Pin the fingerprint in this document (below) and in `README.md` so any signed APK can be verified against the canonical key.

**Canonical SHA-256 fingerprint:** _to be filled in at first generation, Phase 8b._

## 2. Storage — Three-Location Backup Rule

The keystore lives in **three independent locations**, encrypted at rest. **All three must exist before the first production signing.**

| Location                                                                       | Encryption       | Owner                        | Verified |
| ------------------------------------------------------------------------------ | ---------------- | ---------------------------- | -------- |
| 1. 1Password vault `Noor — Production Secrets`                                 | Vault encryption | Personal account             | Phase 8b |
| 2. Offline encrypted USB drive (LUKS / VeraCrypt), stored in a fire-rated safe | LUKS / VeraCrypt | Personal                     | Phase 8b |
| 3. EAS-managed credentials (Expo's hosted vault)                               | EAS vault        | Expo account `badrusiddique` | Phase 8b |

Treat EAS-managed as the **third** backup, not the only one. Expo could change pricing, the account could be suspended, the service could go down. The personal copies are the canon.

The `.keystore` file is in `.gitignore` and **must never be committed**. Pre-commit hook `gitleaks` scans for `*.keystore` and rejects.

---

## 3. Recovery — Lost or Damaged

| Scenario                      | Recovery                                                                                                                                                                                  |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Lost laptop, 1Password intact | Restore from 1Password. Re-establish offline backup within 7 days.                                                                                                                        |
| Lost 1Password access         | Restore from offline encrypted USB. Re-create 1Password entry.                                                                                                                            |
| Lost both personal copies     | Pull from EAS-managed credentials (`eas credentials`). Immediately re-create both personal copies.                                                                                        |
| Lost all three                | **No recovery.** Ship a new app under a new package name (`app.noor.quran.v2`), notify users via release notes + the existing app's update message that the next update is a new install. |

Practice: every 12 months, confirm all three backups are accessible (open 1Password, plug in USB, run `eas credentials` for the project). Log the date of the last verification in `docs/_keystore-verifications.log` (gitignored).

---

## 4. SHA-256 Fingerprint Logging

Every signed release prints its SHA-256 in CI. Compare against the canonical fingerprint:

```sh
unzip -p path/to/noor.apk META-INF/CERT.RSA | keytool -printcert -v | grep "SHA256:"
```

If the fingerprint does not match the canonical one, **do not ship**. Investigate first — could be a wrong keystore picked up by EAS, could be a compromised key.

---

## 5. Compromise — What to Do

If the keystore is suspected compromised (laptop stolen with 1Password unlocked, USB lost without disk encryption verified, EAS account credentials leaked):

1. **Do not panic-rotate.** Google Play does not allow upload-key rotation through the standard upload flow. Rotation requires "Play App Signing key reset" — a human review by Google, takes 1–3 weeks.
2. **Open a Play Console support request** within 24 hours, describe the compromise.
3. **Generate a new upload key** (same `keytool` command, new `keystore` filename, new password).
4. **Submit to Google for upload-key reset.** Ship updates with the old key in the meantime — Google verifies the new one server-side.
5. **Once approved**, swap to the new keystore in EAS credentials. Update this document with the new SHA-256 fingerprint.

The **app signing key** (Play App Signing's master key, held by Google) is rotated server-side automatically. The **upload key** is what we hold, and it is what gets compromised in our threat model.

---

## 6. iOS Signing — Out of Scope

iOS signing is handled by EAS Build via `eas credentials` with Apple Developer Program credentials. The Apple Developer Program account itself is the recovery key — losing it is a different problem (loss of $99/yr account, not "can't update app forever"). Apple supports key rotation through the developer portal.

iOS launch is deferred to v1.1+; this document is updated when iOS ships.

---

## 7. Operational Cadence

- **Every 12 months:** verify all three backups accessible, log to `_keystore-verifications.log`.
- **Every Play Console release:** confirm the fingerprint of the upload AAB matches the canonical fingerprint.
- **On laptop change / OS reinstall:** restore keystore from 1Password to local `~/.android-secrets/` (gitignored, encrypted at rest by FileVault), confirm fingerprint.

This document is part of the QA gate for `v1.0.0` — Stop-Ship row in the matrix until all three backups exist and the fingerprint is logged.
